MAQLUBA

Game Jam Kit — TEAM MAQLUBA

مرجع شامل لمسابقة Rising Devs Godot Game Jam — أدوات، قوالب، مصادر، أفكار، فيديوهات، وروابط مباشرة لكل عضو ودور.

إعداد سريع مخصص لكل دور أصول / Assets صوت أدوات & تطبيقات فيديوهات تعليمية بنك أفكار خريطة عمل 48 ساعة قوالب & GitHub عرض المشروع نصائح احترافية

إعداد سريع — ما جهزناه لكم قبل الانطلاق

  • Git افتحوا مستودع في GitHub — اسم: maqluba-gamejam. ضعوا ملف README يشرح الأدوار وطريقة التشغيل (how-to-run).
  • Godot جهزوا مشروع Godot (prefer Godot 4 إذا تعرفون عليه) مع فولدرات: scenes/ scripts/ assets/.
  • Branch استخدموا فروع بسيطة: main, dev, feature/*.
  • Communication قناة Discord + صوت (VC) + نصية لكل مهمة: #dev #art #sfx #build
  • Backup شاركوا نسخة ZIP على Google Drive/Dropbox تلقائيًا عند كل تسليم مهم.

روابط مفيدة: انشاء Repo Godot.org

Quick SFX Workflow

  1. استخدم Bfxr لصنع SFX أساسي.
  2. نظفها في Audacity → Normalize → Fade out.
  3. أضف reverb بسيط داخل Godot أو engine.

بنك أفكار قابل للتكييف مع أي Theme

1) Puzzle — "توصيل عناصر" (Low scope)

ميكانيك: وصل عناصر متوافقة لإكمال مستوى، صعوبة تدريجية، 10-12 مستوى.

فن: أيقونات بسيطة 32x32 أو 64x64 — يمكن استخدام Kenney assets. صوت: مؤثرات خفيفة. مفيد لـ: فريق صغير، تسليم سريع.

2) Top-down stealth / chase (Medium)

ميكانيك: ممرات، رؤية الأعداء cone of vision، جمع أهداف. يُظهر AI بسيط للأعداء.

مهم للـ programmer (AI), artist (tiles), sound (footsteps).

3) One-button arcade (Minimal scope)

ميكانيك: لعبة تعتمد على زر واحد — مثالي لتقليل الأخطاء وضمان قابلية اللعب. مثال: القفز/التبديل/التحكم في النبضات.

قابل للاندرويد أو للويب بسرعة، مناسب للجيم جام.

4) Narrative micro-game

ميكانيك: تتابع قصصي مع اختيارات قصيرة — تركيز على التقديم والكتابة الجيدة. يناسب لو Theme يميل للسرد.

كل فكرة هنا قابلة للتبسيط أو التعقيد حسب الوقت. ابدأوا بأقل MVP ثم أضيفوا طبقات.

خريطة عمل 48 ساعة — تقسيم وقتي عملي

  1. قبل الانطلاق (جاهز الآن): repo, godot project, assets pack, Bfxr, template scenes.
  2. ساعة 0 - 1: استماع للـTheme + قرار الفكرة — اكتبوا تصميمًا بسيطًا (1 صفحة).
  3. ساعات 1 - 6: بناء الـMVP — حركة اللاعب، input، مشهد واحد قابل للعب.
  4. ساعات 6 - 14: إضافة آليات أساسية، وضع التجارب، رسومات أولية، SFX أولية.
  5. ساعات 14 - 30: توسعة المستويات، polish، تحسينات ميكانيكية، إضافة واجهة بسيطة.
  6. ساعات 30 - 42: اختبار شامل، إصلاح الأخطاء الحرجة، تحسين الأداء، بناء export لWeb/Windows.
  7. ساعات 42 - 48: تحضير العرض التقديمي، فيديو قصير للـ gameplay (30-60s)، رفع الملفات وتسليمها.

قانون ذكي: في اليوم الأول اجعلوا الهدف "قابل للعب" وعملية، وفي اليوم الثاني اجعلوها "ممتعة" — polish أقل لكن تأثير أكبر.

عرض المشروع — ماذا تجهزون للـSubmission

  • لعبة قابلة للتشغيل (export) — Web HTML5 أو Windows build.
  • ملف README يتضمن: وصف اللعبة، كيفية اللعب، دور كل عضو، قائمة assets & تراخيص.
  • فيديو 30–60 ثانية يعرض الفكرة الأساسية وآلية اللعب (رفع على YouTube أو Google Drive).
  • شرائح عرض 3-5 صفحات: المشكلة، الحل (اللعبة)، آليات اللعب، ماذا تم إنجازه في المسابقة.

روابط مهمة للعرض: Canva، Screencast-O-Matic

نصائح احترافية + أخطاء شائعة

  • لا تفرطوا في الـscope. إنقاذ الوقت أهم من إضافة جميع المزايا.
  • نسخ احتياطي عند كل ساعة عمل مهمة. لا تعتمدوا على جهاز واحد.
  • اختبروا على جهاز آخر. Web export سريع للاختبار.
  • وثقوا التغييرات المهمة. سجلوا ما تم إنجازه في كل commit message.
  • إذا واجهتم حطأ كبير: عودوا لنسخة مستقرة ولا تحاولوا إصلاح كل شيء مرة واحدة.
